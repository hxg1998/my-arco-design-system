# Component Rules

本文档定义系统级组件的使用规则。  
默认基于 **Arco Design React** 封装或使用，不重复造基础组件。  
所有组件必须遵循以下原则：

1. 优先复用已有组件，不直接在页面里临时拼样式。
2. 视觉规则统一遵循 `docs/design-tokens.md`。
3. 页面结构统一遵循 `docs/page-patterns.md`。
4. 如现有组件不能满足需求，先说明缺失项，再扩展。
5. 所有组件都必须考虑基础状态：默认、禁用、加载、空态、错误态（按需适用）。

---

## Button

### 用途
用于触发明确操作，如保存、提交、确认、取消、删除、下载、跳转。

### 推荐变体
- Primary Button：主操作
- Secondary Button：次要操作
- Text / Link Button：弱操作
- Danger Button：危险操作

### 何时使用
- 页面中最重要的单个操作
- 表单提交
- 弹窗确认
- 表格行操作
- 工具栏操作

### 禁止使用
- 一个区域出现多个“主按钮”争夺视觉焦点
- 使用按钮模拟纯文本说明
- 仅靠颜色区分意义但没有文案区分
- 页面中随意定义新的按钮样式

### 样式规则
- 高度默认：40px
- 圆角：使用 `radiusMedium`
- 左右内边距：使用 `spaceLg`
- 字重：500
- 按钮内图标与文字间距：使用 `spaceSm`
- 同一行按钮之间间距：使用 `spaceSm` 或 `spaceMd`

### 交互规则
- 主按钮在一个操作区最多 1 个
- 危险操作必须有明确文案，不仅靠颜色提示
- 提交类按钮在请求中必须进入 loading 状态
- 禁用状态必须不可点击且视觉上可识别

### 状态
- default
- hover
- active
- disabled
- loading

### 技术规则
- 默认基于 Arco Button 封装
- 页面中的主操作优先使用 `PrimaryButton`
- 不允许页面直接覆盖主按钮颜色、圆角、字号
- 如需新增尺寸或变体，应先补充到设计规范再使用

## Input

### 用途
用于输入单行文本，如名称、标题、关键词、编号、邮箱、手机号等。

### 何时使用
- 表单输入
- 搜索框
- 筛选条件
- 简短文本录入

### 禁止使用
- 输入较长正文时使用单行 Input
- 在没有标签或占位说明的情况下直接使用
- 把占位符当成唯一字段说明

### 样式规则
- 高度默认：40px
- 圆角：使用 `radiusMedium`
- 内边距：使用 `spaceMd`
- 字号：Body
- 边框颜色、聚焦色、错误色统一使用 token
- 宽度优先按布局系统控制，不在页面中写死大量固定宽度

### 交互规则
- 必填项必须明确标识
- 有格式要求时应提供示例或校验提示
- 错误提示应贴近输入框出现
- 搜索类 Input 可带前缀图标或清空按钮

### 状态
- default
- focus
- filled
- disabled
- error
- readonly

### 技术规则
- 默认基于 Arco Input
- 搜索场景优先统一为 `SearchInput` 封装
- 不允许不同页面随意定义不同边框、不同高度的输入框
- 复杂输入需求优先看 Arco 是否已有现成能力，不重复封装基础交互

## Select

### 用途
用于从预定义选项中选择一个或多个值。

### 何时使用
- 状态筛选
- 分类选择
- 单选 / 多选枚举值
- 表单中的有限选项录入

### 禁止使用
- 选项很少但更适合单选按钮时仍强行使用 Select
- 选项极多但没有搜索能力
- 未明确区分单选和多选交互
- 文案含糊，用户无法理解选项差异

### 样式规则
- 高度默认：40px
- 圆角：使用 `radiusMedium`
- 宽度与 Input 保持一致规则
- 多选标签间距使用 `spaceXs` / `spaceSm`
- 下拉面板圆角、阴影统一使用 token

### 交互规则
- 选项超过 8 个时优先支持搜索
- 多选必须支持已选项清晰展示
- 默认值与占位文案要区分清楚
- 禁用选项必须可识别
- 远程搜索时需处理 loading 与空结果

### 状态
- default
- open
- selected
- disabled
- error
- loading
- empty

### 技术规则
- 默认基于 Arco Select
- 单选、多选、可搜索要明确配置
- 页面中相同业务含义的 Select 应保持相同文案和选项顺序
- 不允许页面自行拼装“伪下拉框”

## Card

### 用途
用于承载一组相关信息或功能模块，是列表页、详情页、表单页中的基础容器。

### 何时使用
- 信息分组
- 表单分区
- 数据摘要
- 内容块容器
- 列表项容器

### 禁止使用
- 卡片层级过深，造成嵌套过重
- 同一页面中出现多种风格不一致的卡片
- 仅为加边框而无信息结构意义时滥用 Card

### 样式规则
- 圆角：使用 `radiusLarge`
- 内边距：使用 `spaceLg` 或 `spaceXl`
- 卡片标题与内容区间距：使用 `spaceLg`
- 边框、阴影、背景统一遵循 token
- 卡片之间纵向间距：使用 `spaceLg` 或 `spaceXl`

### 交互规则
- 标题区可带操作，但操作数量不宜过多
- 可点击卡片必须有明确 hover 反馈
- 卡片内容优先保持单一职责，不混杂过多操作

### 状态
- default
- hover（可点击时）
- loading
- empty

### 技术规则
- 默认使用 `AppCard` 作为系统级统一封装
- 页面中避免直接使用多种不同 padding 的 Card
- 卡片头部、内容区、底部操作区结构应保持统一
- 若出现“统计卡片”“信息卡片”“配置卡片”等固定模式，应进一步沉淀成专用封装

---

## Modal

### 用途
用于承载需要用户中断当前流程并立即处理的任务。

### 何时使用
- 确认操作
- 小型表单
- 风险提示
- 临时信息查看
- 必须在当前上下文快速完成的任务

### 禁止使用
- 内容过长、结构复杂时继续使用 Modal
- 需要连续编辑复杂内容时使用 Modal
- 把本应独立页面的流程塞进 Modal

### 样式规则
- 宽度按内容复杂度分级，不随意自定义
- 标题、内容、底部操作区层级清晰
- 内边距统一遵循 token
- 底部按钮排列统一：主操作在右侧，取消操作靠近主操作

### 交互规则
- 打开 Modal 后焦点应进入弹层
- 关闭方式明确：右上角关闭、取消按钮、遮罩关闭（按场景配置）
- 危险确认弹窗必须明确结果和影响
- 提交时必须支持 loading
- 关闭前如有未保存内容，应有提醒机制

### 状态
- default
- loading
- error
- confirm
- disabled action

### 技术规则
- 默认基于 Arco Modal
- 只承载轻量任务，不放复杂长表单
- 页面对话框类型保持统一，不允许每个页面单独定义弹窗布局
- 标准确认弹窗优先使用统一封装

---

## Drawer

### 用途
用于承载较复杂但仍希望保留页面上下文的编辑或查看任务。

### 何时使用
- 详情查看
- 中等复杂度表单编辑
- 分步配置
- 较长内容但不值得跳转独立页面

### 禁止使用
- 简单确认操作时使用 Drawer
- 内容特别重、流程特别长时仍强行放 Drawer
- 一个页面同时打开多层 Drawer

### 样式规则
- 宽度按任务复杂度分级
- 头部固定展示标题与关闭操作
- 内容区滚动，底部操作区可固定
- 内边距遵循 token

### 交互规则
- 打开后保留页面上下文
- 操作区始终可见，避免用户滚动到底才看到保存按钮
- 有未保存内容时关闭需提醒
- 加载、空态、错误态需明确

### 状态
- default
- loading
- empty
- error
- editing

### 技术规则
- 默认基于 Arco Drawer
- 复杂编辑优先用 Drawer，不优先堆到 Modal 中
- 同类业务的 Drawer 宽度和结构保持一致
- 如出现固定布局，应沉淀为统一 `FormDrawer` / `DetailDrawer`

---

## Table

### 用途
用于展示结构化列表数据，是后台和管理端的核心组件之一。

### 何时使用
- 列表页
- 数据检索结果
- 可排序、可筛选的数据集合
- 行级操作场景

### 禁止使用
- 数据量很小但更适合卡片列表时仍强行用 Table
- 列太多导致无法阅读
- 行操作过多造成拥挤
- 表格中嵌入复杂交互表单造成可读性差

### 样式规则
- 表头样式统一
- 行高统一，不同页面不随意变化
- 单元格内边距统一遵循 token
- 操作列对齐方式统一
- 表格与搜索区、工具栏、分页区间距统一

### 交互规则
- 必须明确 loading、empty、error 状态
- 行操作控制在 2 到 3 个主操作内，其余收纳
- 支持排序、筛选时要可见且可理解
- 长文本需处理截断、换行或 tooltip
- 批量操作必须搭配明确的选中反馈

### 状态
- default
- loading
- empty
- error
- selected
- sorting
- filtering

### 技术规则
- 默认基于 Arco Table
- 列表页优先配合 `TableToolbar`、`PageHeader` 使用
- 相同业务表格的列顺序、状态文案、操作文案尽量统一
- 复杂表格能力优先基于现有封装扩展，不直接在页面中临时拼大量逻辑

---

## Form

### 用途
用于录入、编辑、提交结构化信息。

### 何时使用
- 新建
- 编辑
- 筛选
- 配置管理
- 分步录入

### 禁止使用
- 没有字段分组的大型表单直接堆叠
- 校验规则不明确
- 必填、选填、错误提示不清晰
- 页面中同类表单布局不一致

### 样式规则
- 表单项间距统一使用 token
- Label 与控件间距统一
- 单列表单优先，复杂场景再分栏
- 分组表单优先使用 Card 或 Section 分隔
- 底部操作区和内容区间距明确

### 交互规则
- 必填项必须标记
- 提交前后要有明确反馈
- 校验错误就近展示
- 提交按钮在加载中不可重复提交
- 编辑表单离开页面或关闭容器时，如有未保存更改需提醒

### 状态
- default
- validating
- error
- disabled
- readonly
- submitting

### 技术规则
- 默认基于 Arco Form
- 系统级表单优先通过 `AppForm` 统一布局和校验行为
- 同类表单字段顺序、命名、说明文案应保持一致
- 复杂表单应拆分为多个逻辑区块，不直接堆满一屏

---

## Tabs

### 用途
用于在同一信息层级下切换不同内容区域。

### 何时使用
- 详情页多个内容区切换
- 配置页不同模块切换
- 列表页多状态切换
- 同层级内容分类

### 禁止使用
- 把层级不同的内容硬塞进 Tabs
- Tab 数量过多导致阅读困难
- 一个页面内嵌套多层 Tabs
- Tab 文案过长

### 样式规则
- Tab 文案简洁明确
- 激活态使用品牌主色或统一高亮规则
- 内容区与 Tabs 间距统一
- 不同页面使用相同风格的 Tabs 样式

### 交互规则
- 默认激活项必须符合业务优先级
- 切换时避免不必要的整页跳动
- 如果切换会丢失输入，应有提示
- Tab 数量过多时优先考虑分组或其他导航方式

### 状态
- default
- active
- hover
- disabled

### 技术规则
- 默认基于 Arco Tabs
- 详情页优先统一为标准 Tabs 布局
- 不允许页面中随意定制不同风格的激活线、字号、间距
- 如业务固定出现“概览 / 配置 / 历史”等模式，应沉淀为统一页面模板

---

## Tag

### 用途
用于标识状态、分类、属性、标签信息。

### 何时使用
- 状态展示
- 类型标识
- 筛选结果标签
- 多值可视化展示

### 禁止使用
- 用 Tag 承载主要操作
- 用过多颜色表达无层级的信息
- 同类状态在不同页面颜色不一致
- 文案过长导致视觉噪音

### 样式规则
- 尺寸统一，小而清晰
- 圆角、边框、背景统一遵循 token
- 状态颜色必须在系统内固定映射
- Tag 间距使用 `spaceXs` / `spaceSm`

### 交互规则
- 纯展示型 Tag 不应看起来像按钮
- 可关闭 Tag 必须有明确关闭反馈
- 状态 Tag 文案要简洁，不用长句
- 同类状态使用固定颜色，如成功/警告/失败/进行中

### 状态
- default
- colored
- closable
- disabled（按需）

### 技术规则
- 默认基于 Arco Tag
- 业务状态色必须统一映射，不允许页面自行定义
- 如存在高频业务状态，应沉淀为统一 `StatusTag`
- 状态文案要在系统内统一命名，例如“成功 / 失败 / 进行中 / 已暂停”

## PageHeader

### 用途
用于页面顶部的信息和操作入口，承载标题、描述、面包屑、主要操作。

### 何时使用
- 列表页顶部
- 详情页顶部
- 表单页顶部
- 配置页顶部

### 禁止使用
- 页面头部结构每页都不同
- 页面标题和主操作随意摆放
- 大量说明文字堆在头部
- 同类页面的标题层级不一致

### 样式规则
- 标题字号使用 Page Title
- 标题与描述间距使用 `spaceXs` / `spaceSm`
- 头部与正文区间距使用 `spaceLg` 或 `spaceXl`
- 操作区按钮排列统一
- 面包屑、标题、描述、操作的层级清晰

### 结构建议
标准 `PageHeader` 可包含：
- Breadcrumb（按需）
- Title
- Description（按需）
- Extra Actions
- Tabs（按需，不建议默认混放）

### 交互规则
- 页面主操作放在头部右侧
- 主操作最多 1 个
- 次级操作进行分组或收纳
- 标题需准确表达当前页面目标
- 详情页返回路径应明确

### 状态
- default
- with description
- with actions
- with breadcrumb
- loading skeleton（按需）

### 技术规则
- 页面统一使用 `PageHeader` 封装
- 列表页、表单页、详情页头部结构保持一致规则
- 不允许在页面中直接手写多套不同标题区样式
- 如某类页面头部长期固定，应进一步抽象为专用页面模板

## 通用补充规则

### 命名规则
- 组件名使用清晰语义，不使用 `XXXNew`、`XXXV2`、`TempXXX`
- 状态类组件优先统一命名，例如 `PrimaryButton`、`StatusTag`、`FormDrawer`

### 样式规则
- 页面中禁止随意写新的颜色、字号、圆角、阴影
- 所有样式优先引用 token
- 不同页面的相同组件必须保持同一视觉规则

### 扩展规则
当现有组件不能满足需求时，按以下顺序处理：
1. 先检查 Arco 原生组件是否已有能力
2. 再检查系统是否已有封装组件
3. 如仍不满足，先补充规范文档
4. 最后再新增组件或变体

### AI 开发规则
每次生成页面或组件时，应先检查：
1. 是否优先使用了 Arco 原生组件
2. 是否优先复用了系统封装组件
3. 是否符合 design-tokens
4. 是否符合 page-patterns
5. 是否补全了 loading / empty / error / disabled 等状态
